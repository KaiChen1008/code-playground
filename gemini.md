# GEMINI.md - code-playground

This project is a self-hosted code execution platform (similar to ideone.com), supporting Go, Python, and JavaScript. It serves as a reference for a full-stack, Clean Architecture implementation in Go.

## 🏗 Project Architecture

The project follows **Clean Architecture** principles to ensure separation of concerns and testability:

- `api/`: OpenAPI/Swagger specifications (Source of Truth for API).
- `cmd/server/`: Main application entry point.
  - `delivery/`: Outer layer; handles routing (Gin), requests/responses, and middleware.
  - `usecase/`: Orchestrates business logic, interacts with repositories, and manages concurrency limits (semaphores).
  - `domain/`: Business entities and central interfaces (`Repository`, `Usecase`, `CodeRunner`).
    - `models/`: Data models generated by `go-swagger`.
  - `repository/`: Data access layer (e.g., file-based storage in `./data`).
- `pkg/`: Reusable internal libraries.
  - `config/`: Configuration management (Viper).
  - `errors/`: Custom error wrapping for consistent error handling.
  - `runner/`: Code execution logic using `testcontainers-go`.
- `ui/`: Frontend assets (Monaco editor, Vanilla CSS, IG-style theme).
- `data/`: Local persistent storage for snippets.

## 🛠 Tech Stack & Tooling

- **Backend**: Go (Gin Gonic)
- **API Strategy**: Spec-first development. Use `api/swagger.yaml` and `go-swagger` for model generation.
- **Frontend**: Vanilla JS/CSS + Monaco Editor.
- **Infrastructure**: Docker & Docker Compose.
- **Testing**: `testify` for assertions; `testcontainers` for isolated code execution environments.
- **Logging**: `logrus` with structured logging.
- **Scripting**: Python scripts managed by `uv`.

## 📏 Engineering Standards

### Coding Style

- **Go**: Follow standard `go fmt` and `go vet`. Use `go mod tidy` before any commit.
- **Imports**: Group and order imports in this sequence: 1. Built-in (standard library), 2. 3rd-party libraries, 3. Project-local packages. Separate groups with a blank line.
- **Errors**: Always use the custom wrappers in `pkg/errors` to maintain stack traces and context.
- **Naming**: Use idiomatic Go naming. Prefer `New` for the primary implementation constructor in its package (e.g., `usecase.New`). For specific implementations in the repository layer, use `New<Type>Repo` (e.g., `repository.NewFileRepo`).

### Development Workflow

- **API Changes**: Update `api/swagger.yaml` first, then run `make swagger` to regenerate models.
- **Configuration**: All settings must be defined in `config.yaml` and accessed via `pkg/config`. **Note:** Do not pass the entire configuration object to each struct; pass only the specifically required fields to constructors.
- **Testing**:
  - Every repository and usecase should have a corresponding `_test.go` file.
  - Use `testify/assert` or `testify/require`.
  - Mocking should be used for usecase unit tests to isolate logic.

## 🚀 Common Commands

- `make build`: Compiles the server binary.
- `make run`: Builds and executes the server locally.
- `make test`: Runs the full test suite.
- `make up`: Starts the entire stack (API + Infrastructure) via Docker Compose.
- `make swagger`: Regenerates the `domain` models from the OpenAPI spec.

## 🛑 Quality Gate (Before Stop)

1. Ensure `go mod tidy` has been executed.
2. Run `go vet ./...` to catch common mistakes.
3. Verify that all tests pass: `go test ./...`.
4. Ensure no hardcoded secrets or local paths are present in the code.
